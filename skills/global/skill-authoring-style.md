# Skills 编写风格规范

用于统一后续新增 Skill 的结构、命名、内容深度与可执行性，避免风格漂移。

## 适用范围

- 当你让 AI “新增 skill / 补充 skill / 重写 skill”时
- 当 AI 需要把项目经验沉淀为可复用技能文档时

## 文件位置规范

1. 全局通用能力放：`skills/global/`
2. 前端通用能力放：`skills/front/`
3. 前端 admin 专属能力放：`skills/front/admin/`
4. 后端能力放：`skills/backend/`（后续可按实际项目创建）

## 命名规范

1. 文件名使用小写中划线：`xxx-yyy.md`
2. 文件名应表达能力，不要过度抽象
3. 一个文件只讲一个主能力，避免“大全式”堆砌

## 标题与结构规范

每个 Skill 建议包含以下章节（按需增减，但顺序尽量一致）：

1. `# <Skill 名称>`
2. `## 场景与目标`
3. `## 目录/文件位置`
4. `## 核心依赖或组件`
5. `## 操作步骤`
6. `## 常用命令`
7. `## 注意事项`

## 内容质量规范

1. 结论必须可执行，避免空泛描述
2. 命令优先使用本项目约定（如 `vp run ...`）
3. 代码示例必须贴近当前仓库真实写法
4. 路径必须写项目相对路径，且与实际文件一致
5. 若存在已弃用方案，要明确标注“不要再用”

## 低 Token 编写与执行规范

用于降低 AI 在“读项目 + 改代码 + 验证”过程中的上下文与工具调用开销。

1. 先缩小范围再读取文件：优先 `Glob`/`Grep` 定位，最后才 `Read`
2. 单次只读“必要文件”：
   - 必读：当前改动文件、直接依赖文件、路由/入口文件
   - 非必读：生成文件（如 `routeTree.gen.ts`）、无关模块、超长文档
3. 搜索优先返回路径而非全文：
   - `Grep` 默认使用 `files_with_matches`
   - 仅在确认目标后再切 `content`
4. 控制读取体量：
   - 长文件先按 `offset + limit` 分段读取
   - 避免一次性读取整个大型文件
5. 批量并行读取要有上限：
   - 并行读取建议 `3~5` 个文件
   - 超过上限先做二次筛选
6. 避免重复探索：
   - 同一信息源最多读取一次，后续复用已得结论
   - 仅在用户要求或证据不足时追加检索
7. 输出遵循“结论先行”：
   - 先给修改结果与路径
   - 再补少量关键依据，避免粘贴大段无关代码
8. 校验采用“最小充分”策略：
   - 仅在改动完成后统一执行必要校验
   - 不在中途反复跑全量命令

## 与项目规则联动

1. 新增 Skill 后，需要同步更新 `AGENTS.md` 的 Skills 列表
2. 若涉及工作方式变化，要同步更新：
   - `AI_WORK_LOG.md`
   - `AI_GIT_LOG.md`
3. 若 Skill 影响需求执行方式，可补充到 `AI_REQUIREMENTS.md`

## Skill 模板

````md
# <Skill 名称>

## 场景与目标
- 适用场景：<何时使用>
- 目标：<解决什么问题>

## 目录/文件位置
- `<path/to/file-or-dir>`

## 核心依赖或组件
- `<library/component>`：<作用>

## 操作步骤
1. <步骤1>
2. <步骤2>
3. <步骤3>

## 常用命令
```bash
<command>
```

## 注意事项
1. <注意点1>
2. <注意点2>
````
